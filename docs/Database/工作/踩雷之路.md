## 前言

本篇用于记录工作中项目的踩雷之坑，排雷过程，以及总结。

## 1.重复新增设备

### 1.1 复现场景

用户高频率添加设备，很多请求短时间到达后端，第一次的请求标记设备 A 为新增设备，执行批量修改设备（耗时长），此时第二次的请求进来，又标记设备 A 为新增设备，这时候就会造成重复新增设备的问题了

### 1.2 加锁

> - 解决方案：加锁，确保新增设备请求一个一个执行
> - 锁的类型：Redisson 的公平锁
> - 锁的对象：add*device_lock*用户名

- 依赖：

```xml
<!-- redisson -->
<dependency>
    <groupId>org.redisson</groupId>
    <artifactId>redisson-spring-boot-starter</artifactId>
    <version>3.23.4</version>
</dependency>
```

- Reddsion 配置类

```java
@Configuration
public class RedissonConfig {

    @Value("${spring.data.redis.host}")
    private String host;

    @Value("${spring.data.redis.port}")
    private String port;

    @Value("${spring.data.redis.password}")
    private String password;

    @Bean
    public RedissonClient redissonClient() {
        Config config = new Config();
        String address = "redis://" + host + ":" + port;
        config.useSingleServer()
                .setAddress(address)
                .setPassword(password)
                .setConnectionPoolSize(50)
                .setIdleConnectionTimeout(10000)
                .setConnectTimeout(3000)
                .setTimeout(3000)
                .setDatabase(0);

        return Redisson.create(config);
    }
}
```

- 使用 Redisson 的公平锁

```java
@Autowired
private RedissonClient redissonClient;

@Transactional(rollbackFor = Exception.class)
@Override
public void addDevice(JSONObject json) {

    String key = "add_device_lock_" + getUserId();
    // 加锁，没有获取到锁的会被阻塞
    RLock fairLock = redissonClient.getFairLock(key);
    try{
        log.info("时间：{},线程：{},获取到锁：{}", LocalDateTime.now(), Thread.currentThread().getName(), key);
        final Long userId = getUserId();
		final String all = "全部";
        // 批量查询设备
        // 批量插入、修改设备
    }catch (Exception e) {
        log.error("批量添加设备出错：{}", e.getMessage());
    } finally {
        // 解锁
        fairLock.unlock();
        log.info("时间：{},线程：{},释放锁：{}", LocalDateTime.now(), Thread.currentThread().getName(), key);
    }
}
```

## 2.事务提交在锁释放之后

### 2.1 问题

在对 “重复新增设备” 的 bug 进行测试时，发现还会出现重复新增设备的问题，通过解锁和获取锁的日志时间发现，上一个线程刚解锁，下一个线程在极短时间内就获取锁了。

- 问题：上一个线程还没有提交事务，就释放锁了
- 思路：确保事务提交在锁释放之后

### 2.2 改造

- addDevice：不加事务注解 @Transactional

```java
@Override
public void addDevice(JSONObject json) {
    String key = "add_device_lock_" + getUserId();

    RLock fairLock = redissonClient.getFairLock(key);
    // 加锁
    fairLock.lock();
    try {
        log.info("时间：{},线程：{},获取到锁：{}", LocalDateTime.now(), Thread.currentThread().getName(), key);
        // 获取当前对象
        DeviceServiceImpl deviceService = SpringUtils.getBean(DeviceServiceImpl.class);
        deviceService.insertDevice(json);
    } catch (Exception e) {
        log.error("批量添加设备出错：{}", e.getMessage());
    } finally {
        // 解锁
        fairLock.unlock();
        log.info("时间：{},线程：{},释放锁：{}", LocalDateTime.now(), Thread.currentThread().getName(), key);
    }
}
```

```sql
UPDATE zet_device d
        SET d.ipaddress = '219.136.131.153',
            d.nation    = '中国',
            d.province  = '广东省',

        WHERE d.device_identifier = 'E5FC08'
```

在到网上查，普通的 update 修改语句是加什么锁？

回答：如果 where 列不是索引列的话加表锁，是索引列加行锁，目前判断是加了表锁（但是为什么日志里面显示的是行锁哩？），导致后台想要更新起的设备数据产生冲突，出现死锁

```sql
ALTER TABLE zetlight-test ADD INDEX index_name (device_identifier);
```

### 3.4 预防死锁

以下是网上总结的预防死锁的经验：

- 缩短事务长度
- 同意加锁顺序，打破循环等待
- 减速锁粒度，用行锁
